Hardcoded vs imported

Most cross-stack wiring in this repo uses CloudFormation outputs (${cf:stack.Output}). These three fields are different: nothing in the leetcode stacks owns or exports them, so there is no ${cf:...} source to import from.

FieldSource todayWhy not dynamic?
workQueueUrl, tables, bus ${cf:leetcode-submissions-bff-...} Owned by upstream stacks; exported on deploy.
ecrRepoUri ${cf:leetcode-worker-images-...} ECR repo is a stack output; only the tag is pinned locally.
workerImageTag Literal default + optional CLI override Tag is chosen at build time, not by CloudFormation.
defaultVpcId, publicSubnetIds Account default VPC (pre-existing AWS resource) No leetcode stack creates the VPC; MVP skips SSM lookup.
Design rule from design-research.md section 10: all cross-stack wiring via CFN outputs; no SSM at MVP. VPC ids are account infrastructure, not stack outputs, so they land as checked-in literals for the one dev account we deploy to.

Why pin workerImageTag?

The task definition references ${ecrRepoUri}:${workerImageTag}. That tag must identify an exact image digest the dispatcher code was tested against.

  • ECR tag immutability — the worker-images repo rejects re-pushing the same tag with different content. Each build gets a fresh dev-<uuid10> tag; config must record which one Fargate pulls.
  • No floating dev-latest in config — a mutable tag hides which code is running. A rogue debug task on dev-latest once stole SQS messages while the official service used a different revision.
  • Deploy = explicit promotion — bump the literal after scripts/build-runners.sh (or CodeBuild), then yarn deploy:workers. Git history shows exactly which image a deploy rolled out.
  • Override still exists — ${opt:workerImageTag, 'dev-99eba5e1e1'} lets CI pass --workerImageTag without editing the file, but the default in repo is never left blank.

Roll-forward workflow

  1. Build and push: bash scripts/build-runners.sh (prints new tag)
  2. Update the default in config.yml (or pass --workerImageTag)
  3. Deploy: yarn deploy:workers — task definition picks up the new image
Deploying worker code without updating the tag leaves Fargate on the old image even if ECR has a newer build under a different tag.

Why hardcode VPC and subnet ids?

Fargate tasks need an awsvpc network: a VPC, subnets, and (here) a public IP so they can reach AWS public endpoints without a NAT gateway.

Not exported by any leetcode stack

Unlike the submissions queue or problems table, the default VPC predates this project. No CloudFormation stack in the deploy order creates VpcId or SubnetIds outputs for workers to import.

No SSM Parameter Store at MVP

The sibling apps (pastebin, url-shortener) also avoid SSM for v1. Alternatives like ${ssm:/leetcode/vpc-id} add a parameter lifecycle, IAM for reads, and a second place to look when debugging deploy failures.

Account- and region-specific

vpc-0a17cc97d2e0e43cb is the default VPC in account 579273601730, region ap-southeast-1. Another account or region would have different ids; hardcoding documents "this is the dev target" explicitly.

Public subnets on purpose (dev MVP)

Comments in config call out AssignPublicIp: ENABLED and subnets with mapPublicIpOnLaunch=true. Workers reach SQS, DynamoDB, EventBridge, ECR, and CloudWatch over the public AWS endpoint hostnames via the internet gateway — no NAT hourly charge while iterating.

How service.yml uses these values

leetcode-workers/serverless/service.yml (excerpt)
WorkerSecurityGroup:
  Properties:
    VpcId: ${self:custom.defaultVpcId}

WorkerService:
  Properties:
    NetworkConfiguration:
      AwsvpcConfiguration:
        AssignPublicIp: ENABLED
        Subnets: ${self:custom.publicSubnetIds}

Wrong VPC or subnet ids fail at task placement time with opaque ECS errors. Keeping literals in config makes "which network" grep-able in one file.

Design doc vs current dev wiring

design-research.md section 12 describes the production-shaped target: private subnets, no NAT, VPC endpoints for SQS/DDB/S3/ECR/Logs/EventBridge. That is the v2 hardening path.

Today's hardcoded public subnets are a deliberate dev shortcut: fewer moving parts, lower cost, faster bring-up. When networking hardens, expect new stack outputs or a dedicated leetcode-network stack — not magic discovery of the default VPC.

When to stop hardcoding

Valuev2 approach
workerImageTag CI sets --workerImageTag from the build artifact; prd may pin semver tags. Default in repo stays a known-good dev tag.
defaultVpcId Export from a leetcode-network (or similar) stack; import via ${cf:...} like other cross-stack refs.
publicSubnetIds Replace with private subnet outputs + VPC endpoint stack; remove public IP assignment on tasks.

What stays hardcoded even in v2

The pattern of pinning deploy inputs in git does not go away — only the source changes:

  • Image tag remains an explicit promotion step (never an implicit :latest in production).
  • Network ids move from literals to stack outputs, but workers still read them from config.yml — not runtime discovery inside the container.
Hardcoding here means explicit, reviewable deploy inputs, not "never use CloudFormation imports."
