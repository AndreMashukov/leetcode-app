Submission E2E Fix

Why submissions stuck in RUNNING after the workers stack deployed — and what fixed it.

Resolved 2026-06-25 Verified ACCEPTED path Image dev-99eba5e1e1

Symptom

Synthetic SQS messages exercised the worker pipeline: submissions transitioned PENDING → RUNNING within seconds but never reached a terminal status (ACCEPTED, WRONG_ANSWER, etc.). CloudWatch on the official workers log group showed startup lines only — no result lines.

The runner subprocess worked when invoked manually from a debug ECS task, so the failure was in the dispatcher’s post-run DDB write path (or something intercepting queue messages before the official worker could finish).

What actually happened

flowchart TD
  SQS[work-queue message] --> Rogue[test-workers-cluster task\nold dev-latest image\nno CloudWatch logs]
  SQS --> Official[leetcode-workers-dev task\ndev-99eba5e1e1 image]
  Rogue --> Claim[Conditional claim\nPENDING to RUNNING]
  Claim --> Run[Runner succeeds\npassedCount equals totalCount]
  Run --> Bug[Double-wrapped resultSummary\nParamValidationError]
  Bug --> Stuck[Row stays RUNNING\nSQS message deleted or retried]
  Official --> Skip[Retry sees RUNNING\nnot PENDING — skipping]
The initial debug hypothesis — stderr buffering hiding logs — was wrong. Claim and skip lines appeared in CloudWatch once the correct worker handled retries. The winning worker on first attempt had no log group.

Blocker 1 — CloudWatch log group mismatch (pre-fix deploy)

Before e2e testing, Fargate tasks failed during init because service.yml pointed the awslogs driver at leetcode-workers-dev while cluster.yml created /aws/ecs/leetcode-workers-dev. Fixed by aligning the full path in the task definition.

Blocker 2 — Double-wrapped resultSummary (code bug)

On the happy path, _set_terminal wrote:

leetcode-workers/runs/dispatcher.py (before)bug
values[":rs"] = {"M": _marshall(result_summary)}

_marshall(dict) already returns {"M": {...}}, so the expression became a nested M.M shape. Boto3 rejected it client-side with ParamValidationError before DynamoDB was contacted.

Failure modes that set result_summary=None (compile/runtime errors, missing problem row) still worked — which made the bug look like a hang on successful submissions only.

Stuck rows showed workerId=worker-64cc0bd9 with no matching log stream in the official log group — the smoking gun for a second consumer.

Code change

leetcode-workers/runs/dispatcher.pyfixed
if result_summary is not None:
    set_clauses.append("resultSummary = :rs")
    # _marshall(dict) already returns {"M": {...}} — do not wrap again.
    values[":rs"] = _marshall(result_summary)
