## A. The shape of what changed

Compared to the version I drafted, three things are now different:

### A1. Threat model is now explicitly scoped (was implicit before)

The current version has a clear **"Fargate v1 is not a full untrusted-code
sandbox"** statement in four places:

- §0: "v1 is a **Fargate MVP sandbox** for internal/trusted users; true
  per-submission container isolation (custom seccomp / gVisor / Docker-in-Docker
  style controls) moves to an ECS-on-EC2 worker fleet if the platform accepts
  arbitrary untrusted public code."
- §3 last paragraph: "Fargate v1 also does **not** satisfy the full
  custom-seccomp / Docker-in-Docker sandbox promised by the generic design
  doc..."
- §5.3 / §5.4: split the v1 controls from the production ECS-on-EC2 path.
  §5.4 is the production path with the four Docker flags from example 2 of
  the source spec.
- §10.1: "Worker runtime — Fargate MVP, ECS-on-EC2 for public untrusted
  code."

**Why this matters:** the source spec is uncompromising — "Network access
is completely disabled. The filesystem is mounted as read-only. System calls
are restricted using seccomp profiles" (lines 81–84). Fargate cannot do
three of those four. The doc is honest about that and lays down a clear
v1 / v2 boundary.

### A2. Worker image is one combined Python+Node image (was two services)

§4 `leetcode-workers` and §10.4 both say **one combined ECR image with
Python 3 + Node 20, dispatched by `language` field** rather than two
separate Fargate services.

The trade-off is documented:
- **Pro:** one task def, one service, one SQS queue, halved warm-pool
  cost, simpler auto-scaling.
- **Con:** ~400 MB image instead of two ~200 MB images. Acceptable.

§10.4 says v2 switches to "two-image task with one runner per container,
dispatched at the queue level (use a second SQS queue per language)" if
compiled languages are added.

### A3. §7 cost posture has honest Fargate-on-demand numbers

§7 cost table shows **$180/month on-demand, $55/month Spot** for the
10-task idle floor (not the $300 figure I had originally written). The
ECR row is now "1 combined runner image" not "2 ECR repos". A new
"excludes VPC endpoint hourly charges" caveat is included. The §7 total
is **~$200–230/month on-demand steady-state** (or materially less on Spot).

## B. Internal contradictions / things to flag

### B1. §0 stack count inconsistency (likely harmless)

§0 says "v1 is a **Fargate MVP sandbox** for internal/trusted users" and
proceeds to enumerate 6 stacks in the building-blocks table. §1 has the
same 6-stack count. §9 also has 6. The doc is consistent on the count.

But §0 has a slightly different "what we explicitly do NOT build in v1"
table that uses "Read-replica database" wording, while §1 uses the same
table but with a slightly different framing. Both are present. The reader
sees the same info twice. Not wrong, but verbose.

### B2. §3.3 sandbox table — Lambda row is misleading

§3 has this table:

| Concern          | Lambda limit         | Fargate fit         |
| ---------------- | -------------------- | ------------------- |
| Per-submission runtime | 15 min max    | No limit (practical cap by task timeout, default 30 min) |
| Memory ceiling   | 10 GB                | Up to 120 GB        |
| Ephemeral storage | 10 GB /tmp          | Up to 200 GB volume mount |
| Cold start       | 500ms–2s on first invoke | Warm pool via `desiredCount: 10` (the user's target) |
| Runtime sandbox  | Lambda runtime sandbox; no warm worker process | Fargate task boundary + resource limits in v1; ECS-on-EC2 required for custom seccomp / per-submission containers |
| Language compile | No persistent build cache | Container image bakes the language runtime + any pip/npm cache |
| Cost at steady state | $0.0000166667 per GB-second (Lambda) | $0.04048/vCPU-hour on-demand; ~$0.01012/vCPU-hour on Fargate Spot |

The "Runtime sandbox" row says Lambda = "Lambda runtime sandbox; no warm
worker process". That's the column-1 (Lambda limit) — but the cell is
describing what Fargate **does better**, not what Lambda's limit is.
Slightly garbled. The intended read is "Lambda has a sandbox but no warm
worker process" — and the Fargate fit column says what the better answer
is. Reads fine if you squint.

### B3. §4 redundant ECR bootstrap section — FIXED

§4 originally had two sections that both described the ECR repository:
`leetcode-worker-images` (bootstrap) and `leetcode-workers` (consumer).
The `leetcode-workers` section now correctly delegates ECR ownership to
`leetcode-worker-images` and does not duplicate the resource list.

### B4. §5.2 sandbox code: `setrlimit` references — PARTIALLY FIXED

The Python `apply_user_code_sandbox()` function in §5.2 has all the
sandbox primitives commented out except `RLIMIT_AS` (memory). That
remains true — there is no inner CPU limit applied; only memory.

**What changed:** §3 paragraph 315–320 is now explicit that "There is
no inner CPU `setrlimit` in the v1 sketch; CPU is bounded by the Fargate
task's 0.5 vCPU allocation and the parent timeout."

**What still needs fixing:** §5.3 line 815–816 still reads:

> "Task-level CPU/memory limits (`0.5 vCPU`, `1 GB`) plus inner
> `setrlimit` for user-code memory/time."

The "memory/time" wording is now misleading — there is no inner
setrlimit for time (time is enforced by `subprocess.run(timeout=...)`,
not `RLIMIT_CPU`). The fix is one word: change "memory/time" to
"memory only (time enforced via the parent subprocess timeout, not
setrlimit)". Then §5.3 lines up with §3.

### B5. §3 "Per-task concurrency: 50" — FIXED

§3 originally said "Per-task concurrency: 50 in-flight messages" which
contradicts §5.1's `max=1` runner loop. Now §3 reads:

> "**Backlog scale-out target:** 50 visible messages per running task.
> The v1 runner loop processes one SQS message at a time (`max=1`), so
> this is **not** per-task concurrency."

Clean fix. ✅

### B6. §7 cost posture — FIXED

Originally said ~$300/month for 10 idle Fargate tasks. Real on-demand
is ~$180/month; Spot is ~$55/month. The doc now shows both numbers
explicitly, the ECR row is "1 combined runner image" (was "2 ECR repos"),
and a "VPC endpoint hourly charges" caveat is included. The §7 total
is ~$200–230/month on-demand, materially less on Spot. ✅

### B7. §10.7 Cognito citation — PARTIALLY FIXED

Originally cited "User's answer to question 8: Cognito, one pool" —
but the "one pool" was an inference, not a direct answer. Now §10.7
reads:

> "User's answer to question 8: **Cognito**. The **one shared UserPool**
> is this design's implementation choice so all BFFs use the same JWT
> issuer and audience."

Better — it separates the user fact (chose Cognito) from the design
choice (one shared UserPool across all BFFs). The §10.7 header is
"Auth — Cognito, implemented as one shared UserPool" which is also
clean. ✅

### B8. §10.9 / §4 "code size — 64 KiB cap" — FIXED

§4 now says "(intentionally lower than pastebin because solutions are
small)". §10.9 now says "Matches pastebin's `MAX_SIZE_BYTES = 256 KiB`
constraint style, but intentionally lower because competitive
programming solutions are usually much smaller than paste bodies."
Both are honest. ✅

### B9. §5.2 user code spawn loop has a missing `try/except` — STAYS

§5.2's `run_user_solution_in_sandbox()` shows the per-test-case loop
but does not show the `TimeoutExpired` / `CalledProcessError` handlers

## C. Things I had to look up to verify

### C1. Fargate on-demand vs Spot pricing

Looked up current Fargate pricing for ap-southeast-1. On-demand:
$0.04048/vCPU-hour, $0.004445/GB-hour. Spot discount: ~70% (so
~$0.01214/vCPU-hour).

**Result:** the doc's "~0.01012/vCPU-hour on Fargate Spot" in §3 is
**close but slightly optimistic** (Spot is closer to $0.012, not
$0.010). Also, Spot is interruptible — the doc doesn't say what
happens to in-flight submissions when Fargate reclaims a Spot task.
The §5.1 contract says workers update `status=RUNNING` with a
`workerId` and `startedAt`, so a reclaimed task would leave a
RUNNING row whose `startedAt` is older than max runtime. The §5.1
note says "v1 keeps the simpler no-reclaim behavior and relies on
the DLQ" — meaning a reclaimed task's submission gets DLQ'd after
3 redelivery attempts. That's acceptable, but worth saying
explicitly in §3 that v1's Spot interrupt path = DLQ.

### C2. DDB single-item 400 KB limit

Confirmed: 400 KB is the per-item size limit. The §10.2 reasoning
holds.

### C3. gVisor on Fargate

The §10.1 "ECS-on-EC2 with gVisor/runsc" claim is accurate — Fargate
does not support custom runtimes, so gVisor needs EC2.

### C4. Cognito Hosted UI pricing

Cognito MAU pricing in §7 "free tier covers dev" is correct — first
50K MAUs are free, then tiered.

### C5. ECS target tracking on SQS

Confirmed: ECS does **not** have a predefined `SQSQueueVisibleMessageCount`
target tracking metric. CloudWatch metric math (queue visible / running
task count) is the textbook approach. The doc's §3 step-scaling config
is correct.

### C6. ECR cross-account / same-account private pull

The `leetcode-worker-images` ECR repo and the `leetcode-workers` ECS
service are in the same AWS account (579273601730). ECR private pull
from Fargate in the same account works without cross-account auth.
No issue.
